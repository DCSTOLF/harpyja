# Scout domain

Consolidated, current requirements for Tier 1 (Scout) — Harpyja's self-contained
explorer-loop locator: a general OpenAI-compatible tool-calling model driven over a
bounded, read-only, repo-confined tool suite (grep/glob/read_span/ls/symbols) to a
`submit_citations` terminal action, behind the unchanged `ScoutBackend`/`Locator` seam,
with loopback-only air-gap containment, span normalization, and a typed degradation
taxonomy floored on Tier-0. The FastContext backend it replaced (specs 0005/0007/0011/0012)
is retired and deleted (specs 0024/0025). Each line carries its originating spec(s) as
provenance; superseded text lives in `.archive/scout.md`.

- Scout (Tier 1) is a model-backed locator that drives a general OpenAI-compatible tool-calling model over a bounded read-only exploration loop (repo-confined tools → a citation list) reached through the Gateway on a loopback-only endpoint; it preserves the read-only and air-gap guarantees and is invoked only on explicit opt-in. (specs 0005, 0024)
- Scout sits behind a `ScoutBackend` Protocol (`query + seed spans → list[CodeSpan]`) exposed as a `Locator` and producing `CodeSpan`s with `source_tier = 1`; the orchestrator/formatter never branch on which engine produced a span, and tests inject a fake backend with no live model. (spec 0005)
- Scout hands its backend exactly the whitelist — the bounded read-only explorer tools plus the loopback-enforced Gateway model client — and never a raw base_url, env-derived endpoint, HTTP client/session, or other transport; the backend's egress containment is verified by a network-deny integration test, not an asserted guarantee. (specs 0005, 0024)
- Scout output is model-backed and non-deterministic, so it is not persisted and carries no engine-identity/cache slot. (spec 0005)
- Whenever Scout is invoked it runs its own lightweight Tier-0 seed lookup before the backend call and passes the top-`scout_seed_top_n` spans as hints; the seed-before-backend ordering is contractual, so a missing `rg` surfaces `RipgrepMissingError` first and the backend is never reached. (spec 0005)
- Scout's terminal output is the tool-call-native submit_citations action (strict schema — unknown/extra/diagnosis-shaped fields rejected, the enforceable locator-not-diagnoser guard; no repo-read capability), whose structured refs normalize via the unchanged normalize_spans: a path:start[:end] ref → a spanned CodeSpan and a bare/malformed path → a file-level CodeSpan (no fabricated range), refs normalized/clamped/repo-confined with out-of-repo/nonexistent/over-budget/duplicate refs dropped (bounded by scout_max_citations and scout_max_span_lines), and empty is honest-empty, not a raise. (specs 0011, 0024)
- A Scout call resolves to exactly one of four never-collapsing caller-visible states: (1) success → Scout citations, `tiers_run=[0,1]`; (2) model unavailable with Tier-0 results → Tier-0 citations, `confidence=degraded`, cause-named note; (3) model unavailable, Tier-0 honestly empty → empty + distinct `+no-matches` + cause note; (4) Tier-0 hard precondition absent (e.g. `rg`) → typed error propagates loudly. (spec 0005)
- Degrade notes are stable machine-readable scout-degraded:<cause> identifiers (never free prose), optionally suffixed +no-matches; the explorer's typed causes are model-unreachable, backend-error, loop-turns-exhausted, loop-wallclock-exhausted, and generation-truncated, each a terminal loop state routed to the Tier-0 floor. (specs 0005, 0024, 0028)
- Scout drives Harpyja's own bounded explorer loop — a general OpenAI-compatible tool-calling model over read-only tools to submit_citations, a distinct engine that keeps Tier 1 structurally separate from Tier 2; Scout is never routed through Deep's machinery. (specs 0007, 0024)
- The explorer's default gateway pins settings.lm_model (the same model Deep runs; the gateway's model='local' default 404s on Ollama's tag-routed API), threaded through the canonical build_scout_engine; the FastContext-lineage scout_model setting is NO LONGER the Scout backend model — it survives only as the served Verification-Gate A/B baseline. (spec 0025)
- assert_local(resolved endpoint) fires before any model I/O — complete_with_tools routes through it before any transport touch and ExplorerBackend also calls gateway.assert_local() once before the loop starts; a non-loopback endpoint raises AirGapError and the loop never starts (no model call, no tool call), with zero non-loopback egress proven by a network-deny test. (specs 0007, 0024)
- ScoutUnavailable is raised only for a typed terminal loop failure routed to the Tier-0 floor — never for weak or empty citations (a well-formed empty submit_citations is honest-empty, not a raise); AirGapError and RipgrepMissingError propagate as hard floors, never degrade. (specs 0007, 0024)
- normalize_spans_with_tally returns a per-shape tally (spanned / file-level / dropped) as ScoutTally metadata on the Scout result so every drop is visible and none is silent; the shared tally core is retained backend-neutrally (now populated by the explorer), and the FC-era recovered-count fields are structurally zero. (specs 0011, 0025)
- The explorer exposes EXACTLY five read-only navigation tools — grep, glob, read_span, ls, symbols — each confine_path-guarded, Settings-bounded and repo-confined (out-of-repo scope / ../ traversal / over-budget rejected or clamped), with glob/ls path results normalized to file-level CodeSpan records not raw strings; grep wraps the SAME symbols.ripgrep.RipgrepEngine the Deep search host tool wraps (one bounded ripgrep source of truth, no second grep surface) and an exact-count test pins the tool surface. (specs 0024, 0027, 0030)
- - The explorer injects ZERO repo content pre-loop — build_initial_prompt(query) carries only the task/query plus a tool-usage framing that names EVERY registered tool (the five navigation tools AND the terminal submit_citations, documented as terminal) and the symbols when-to-use (candidate-file → exact definition span; repo-wide by-name lookup when the query's words are ungreppable) — still a small constant independent of repo size (no manifest term; the eager whole-repo map stays removed, push→pull); a structural drift guard (test_initial_prompt_binds_to_registered_tool_surface_single_source, derived from the SAME single source as the schema-vs-dispatch test) FAILS if any registered tool is absent from the prompt, so a new tool is un-shippable without appearing in it — the 0027→0030 silent omission that unadvertised symbols/read_span for 5 specs cannot recur. (specs 0027, 0042)
- The bounded explorer loop dispatches ALL N parallel tool_calls the model emits in one model turn (each answered by its tool_call_id — the OpenAI-correct shape), appends raw tool output to history, and terminates at scout_max_turns or a whole-loop scout_wall_clock_s ceiling (an injected monotonic clock trips it when turns alone would not, so a slow/hung turn cannot wedge); turns_used increments per model_call not per tool_call, and unknown tool names are refused, not dispatched. (specs 0024, 0029)
- A submit_citations call at any position in a parallel batch is terminal — it returns immediately and the remaining calls in that batch are not executed. (spec 0029)
- A non-floor tool exception is recorded as an in-conversation marker tool-call-degraded:execution-error: <Type>: <msg> and the batch continues — it is NOT a terminal ScoutUnavailable cause and is not counted in the report degrade taxonomy; RipgrepMissingError / AirGapError are re-raised (never degraded) inside per-call handling and still propagate to the Tier-0 floor. (spec 0029)
- In-loop self-recovery is deterministic: an exact (tool_name, normalized_args) repeat over scout_loop_repeat_n consecutive no-new-span turns triggers a corrective injection, and history past scout_history_char_cap is truncated dropping ONLY stale navigational chatter — a compact index of any dropped citable span is re-injected so truncation never converts a real find into honest-empty. (spec 0024)
- The explorer surfaces a per-run turns-consumed reading (ExplorerBackend.last_turns_used → ScoutEngine.last_turns_used, reset per run — the migrated 0022 diagnostic) but NEVER branches on turns_used to infer outcome or degrade-kind; the discriminant is LoopResult.outcome / ScoutUnavailable.cause, pinned by an executable source-sweep guard. (specs 0025, 0027)
- The explorer's per-call generation is bounded by explorer_max_tokens (Settings default 2048, also pinned as ExplorerBackend.max_tokens's own field default so a Settings-bypassing construction is still capped) as the primary anti-runaway lever; ModelGateway keeps no max_tokens default of its own, so the Deep-tier path is never capped. CAVEAT (0033): a thinking-by-default served model (qwen3:14b) generates reasoning the gateway silently DROPPED, so the cap has been consumed invisibly by unseen reasoning tokens since 0028 — 0031–0033 live baselines carry invisible-truncation-risk; surfacing reasoning is the spec 0034 fix. (specs 0028, 0033)
- explorer_enable_thinking is a Settings knob: when False the explorer's gateway call carries chat_template_kwargs={'enable_thinking': False}, when True the param is omitted (default True); the explorer_-prefixed knobs are passed ONLY by the explorer, with a Deep-scope guard asserting the Deep-tier outbound call carries neither the cap nor the thinking param. CAVEAT (0033): on the served qwen3:14b (thinks by default) both tested knob settings measured INERT in the 0033 think-experiment — the knob's live effect is UNVERIFIED pending spec 0034's reasoning observability. (specs 0028, 0033)
- A model turn with finish_reason == 'length' is a typed degrade emitting scout-degraded:generation-truncated REGARDLESS of whether a syntactically valid tool_call rode along (a length-truncated response was cut mid-generation and its args may be silently incomplete), checked right after the model call before any tool dispatch. (spec 0028)
- - The symbols tool wraps the existing Tier-0 SymbolRecord index (no new parser, the shared record_to_codespan projection) with `path` OPTIONAL: with a path it returns the file-local definitions as a BARE list[CodeSpan] (shape parity with every other nav tool, so _spans_of registers its spans in seen-span/loop-detection accounting — the pre-0042 nested {'symbols': [...], 'degraded': bool} dict registered ZERO spans and structurally penalized every call, the root of 0040's 0/28 adoption), clamped by scout_symbols_max_entries=400, a ManifestEntry.degraded file falling back to the shared ripgrep engine with the stable ANNOTATION marker `symbols-degraded: '<path>'` PREPENDED (marker first, real spans behind — the 0035 convention's second successful-but-degraded case; model-visible via stringification, never counted as a span); with `name` and no path it does a repo-wide by-name lookup over the records (read-only, repo-confined, no filesystem walk — the NAME-based partial answer to lexical unreachability), ranked exact > prefix > substring with deterministic (path, start_line) ties, clamped by the DISTINCT scout_symbols_repo_max_entries=200; absent/degraded Tier-0 index → the REPLACEMENT marker `symbols-index-unavailable: '<name>'`; a nonexistent path with no records → `symbols-path-not-found: '<path>'` (records WIN over disk absence — the no-new-parser contract); hostile/oversized name → a clipped `symbols-name-invalid` marker; an EMPTY-STRING path is treated as absent, not file-local (small models emit "" for an omitted arg). (specs 0030, 0042)
- ExplorerBackend captures last_trajectory after the loop via build_trajectory_record (mirroring the last_turns_used seam, with the served model threaded from the model call) for postflight trajectory verification, while run() still returns list[CodeSpan]. (spec 0031)
- Every path-discovering explorer tool emits repo-relative paths: RipgrepEngine.search takes an optional per-call repo_root= and re-prefixes parsed scoped paths at the ONE engine seam (the rg invocation is byte-identical), and the explorer grep, the symbols degraded fallback, and Deep search all supply repo_root=repo_path as DATA — the re-prefix logic lives solely in the engine, never per-caller and never by downstream repair. (spec 0033)
- submit_citations returns SubmitResult(spans, submitted, surviving) (submitted = raw ref count, surviving = normalized count); the counts thread as data LoopResult.citations_submitted/surviving → ExplorerBackend → build_trajectory_record → the persisted verifier artifact, so a found-then-dropped (1, 0) is structurally distinguishable from honest-empty (0, 0) rather than hiding inside the empty terminal bucket. (spec 0033)
- The FastContext surface is deleted and stays gone — an executable import-absence / public-name guard rots-false if fastcontext/client/tools re-import or if the FC-era <final_answer> text grammar, _recover_suffix/MIN_TAIL_SEGMENTS suffix recovery, FC error causes, or FC-only Settings fields reappear; suffix recovery in particular stays banned, since the explorer's submitted paths come from real tool output and never needed the prefix hack. (specs 0025, 0033)
- `ExplorerBackend.wrapped_model_call` is a per-turn accumulator appending `{reasoning_chars, completion_tokens, finish_reason}` per model response (reset per run, threaded through `build_trajectory_record` as a parallel `per_turn` list) — the ONLY seam that observes a `finish="length"` FINAL turn, which never enters `model_turns`, so the two lists carry an intrinsic length skew and must not be zipped positionally, making a truncated-by-reasoning turn distinguishable from a content-truncated turn and a clean turn; `derive_think_mode(think, enable_thinking)` records one canonical enum `{default-omitted, native-think-true, native-think-false, chat-template-disabled, unknown}` (native `think` wins on double-set), and as of 0038 the explorer sends `reasoning_effort` ("high" for True / "none" for False) ONLY when `explorer_think` is non-`None` (default omit = outbound request byte-identical, the surviving 0034 pin); the dead top-level `think` field is no longer sent. (specs 0034, 0038)
- The explorer `grep`/`ls` wrappers separate could-not-search from searched-and-found-nothing: a nonexistent scope returns the stable model-visible marker string `grep-scope-not-found: '<scope>'` / `ls-path-not-found: '<path>'` (returned BEFORE any engine delegation, via an `exists()` guard over the non-strict `confine_path` resolve) — non-terminal, NOT a `ScoutUnavailable`, never wearing the 0029 `tool-call-degraded:execution-error:` mislabel, not counted in the degrade report, with `note_navigation` still tripping loop detection on repeated bad scopes and requiring ZERO `explorer_loop.py` changes; the obsolete file-scope `is_dir()` early-return guard is DELETED so an existing-FILE scope DELEGATES to the engine and returns real repo-relative matches (converging `grep` with Deep `search` on the one engine contract), while a searchable scope with zero matches still returns plain `[]` and `ls` on an existing FILE keeps honest-`[]` (spec 0035)

(spec 0038)
