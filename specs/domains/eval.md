# Eval domain

Consolidated, current requirements for the evaluation harness â€” measurement-only,
recommend-only â€” including the dataset formats, the overlap oracle, the
threshold/top-n sweep, SWE-bench adaptation, and degrade-visibility reporting.
Each line carries its originating spec(s) as provenance.

- The eval harness is measurement-only and recommend-only: it observes the real `mode=auto` path without modifying tier/orchestrator/gate/matrix behavior and emits recommendations (e.g. an OQ2 trade-off table) without flipping any `Settings` default â€” default flips are separate follow-up specs citing the evidence. (spec 0009)
- Eval-only knobs (repeated-run count `K`, proximity window, N-floor, catch-rate bar, `degraded_dominated_threshold`) live on a dedicated `EvalConfig` dataclass in `harpyja/eval/`, field-disjoint from the production frozen `Settings`; the only `Settings` interaction is the sweep building grid points via `dataclasses.replace`, never mutation. (spec 0009)
- The eval dataset format is per-case `{query, repo, expected file:line span(s), classification label}` stored as a versioned fixture; the loader rejects malformed cases loudly with a typed, actionable error, never a silent skip. (spec 0009)
- Span-hit accuracy is line-range overlap as the primary metric, with file-level + proximity as a secondary looser metric reported alongside; a single overlap oracle defines correctness and is reused for both gate catch rate (wrong Tier-1 â†’ escalated) and gate false-escalation rate (correct Tier-1 â†’ wrongly escalated). (spec 0009)
- Each measured configuration runs `K` times and the report carries mean + spread per metric; a configuration is recommended over another only when its advantage exceeds the observed run-to-run variance. (spec 0009)
- Sweep mode runs the case set across a `verify_threshold` Ã— `verify_top_n` grid (each point via `dataclasses.replace`, `K` runs per point) and reports the accuracy/escalation/cost trade per point; an N-floor below which results are flagged indicative-only is stated in the report. (spec 0009)
- The runner emits a JSON + human report conforming to a pinned schema (stable per-case and aggregate field names); report artifacts are written to a temp/output directory outside any indexed/target repo (read-only guardrail), and the run path is fully offline (zero non-loopback egress). (spec 0009)
- The SWE-bench Verified dataset adapter uses the standalone localization protocol (no Docker, no patch application, no test execution): it queries the pre-fix repo at `base_commit` and scores predicted citations against patch-derived pre-image hunk spans. Absolute accuracy is treated as contamination-caveated; the load-bearing outputs are relative deltas (sweep and `fast`-vs-`auto`). (spec 0010)
- The dataset pipeline (`harpyja/eval/swebench_eval.py`) separates stages by network posture: `convert` (HuggingFace) â†’ committed portable `swebench_verified.raw.jsonl`; `provision` (may `git clone`) â†’ gitignored machine-local `swebench_verified.resolved.jsonl` with `repo` rewritten to a worktree at `base_commit`; `prune`; and offline `run`/`sweep` subcommands. `convert`/`provision` are dev-time tools outside the runtime air-gap guarantee; `run`/`sweep` are offline. (spec 0010)
- Query classification for SWE-bench cases is derived from patch shape (single file with small total pre-image span â†’ `point`; multi-file or large-span â†’ `broad`), and the driver overrides production routing to that label via the `LocateStack.classifier` injection seam so the gate fires on the point subset; each per-case event records both the patch-shape label and the production `classify_query` label and the aggregate reports their agreement rate, with the OQ2 recommendation flagged low-confidence (deltas-only) below a named agreement floor. (spec 0010)
- Cases whose gold-patch targets are all new files (`--- /dev/null`) are flagged `new_file_only` and excluded from span/file scoring (the excluded count is a durable report field), never scored as a silent zero; a pure-insertion hunk in an existing file is anchored to a concrete one-line span and is scored. The pre-image hunk oracle's context-line inflation biases span-hit upward and is recorded as a tolerance. (spec 0010)
- A per-case-repo driver builds a fresh `LocateStack` per case using that case's worktree as `repo_path` and pools outcomes into the aggregate via the unchanged overlap oracle and recommend scorer; the report schema is extended additively (fields appended last with defaults, `SCHEMA_VERSION` bumped) so both single-run and multi-repo shapes validate, and artifacts are written via `atomic_write_json` outside every case repo. (spec 0010)
- `run_case` threads `mode` so the driver can run the set `mode="fast"` (a Scout-terminal no-climb Tier-1 line where the gate runs informationally) alongside `auto`; the report carries durable metadata fields including protocol identity, excluded/malformed counts, classifier-agreement rate, span-inflation tolerance, contamination caveat, dataset provenance (HF id/split/revision, raw-fixture hash, sample case-ids), and a SUT-observed `production_gate_ran` flag kept distinct from the harness-observed `gate_triggered` probe. (spec 0010)
- The harness reports degrade visibility as first-class machine-readable fields: `scout_degrade_count` and `scout_degrade_rate` (= count / cases_attempted, explicit `null` paired with a zero count when the denominator is zero), `degraded_dominated` (rate > `degraded_dominated_threshold`, provisional eval-only `0.5`) with composable `reliability_notes` stable identifiers, and FC text-ref shape counters `fc_citation_{spanned,filelevel,dropped}_count`. (spec 0011)
- The overlap oracle scores a file-level (line-less) citation as a path-only match recorded distinctly from a line-overlap match (a coarse hit is never a line hit), taking the path-only branch before the line arithmetic; a file-level cited span serializes its line fields as JSON `null` and `validate_report` tolerates null cited lines (gold `expected_spans` stay int-only). (spec 0011)
- Recovery observability is plumbed producerâ†’sink: `build_scout_engine` loads the manifest file list into `ScoutEngine` (empty set for unit fakes / index-not-ready â†’ no recovery), both `eval/runner.py` and `eval/swebench_eval.py` aggregate the per-case `last_tally` counts, and `fc_citation_recovered_{spanned,filelevel}_count` are appended last-with-default (`SCHEMA_VERSION` `0012/1`, later additively bumped to `0013/1` for the `deep_degrade_count`/`deep_degrade_rate` twins and union-based `degraded_dominated`, validator accepts legacy and new shapes). (spec 0012; extended spec 0014)
- Every typed-degrade floor reports a paired `<tier>_degrade_count`/`<tier>_degrade_rate` twin (additive, appended last-with-defaults, explicit `null` paired with a zero count on a zero denominator) **and** feeds `degraded_dominated`, which keys off the **union** of all tiers' per-case degrades â€” a case counted **once** even when multiple tiers floor â€” with the per-tier rates kept separate for attribution; a shared `_has_degrade_note` predicate backs `_is_scout_degraded`/`_is_deep_degraded`. Deep's `deep_degrade_count`/`deep_degrade_rate` (report schema `0013/1`) is the first additional instance after Scout. (spec 0014)
- `cmd_provision` (`eval/swebench_eval.py`) resolves `--work-dir` to an absolute path (`Path(args.work_dir).resolve()`) before `git worktree add`, so a relative work-dir produces worktrees whose real location matches the resolved `repo` path recorded in the machine-local fixture â€” a relative dir otherwise created trees under the clone cwd while the fixture recorded `wt.resolve()`, so every resolved repo path 404'd. (spec 0015)
- The `run`/`sweep` parsers expose model overrides via the shared `_add_model_flags` group: `--scout-model` (â†’ `scout_model`) and the canonical `--deep-model` (â†’ `lm_model`), with `--lm-model` retained as a deprecated back-compat alias; the two Deep flags get distinct argparse dests reconciled in `_settings_from_args` so `--deep-model` wins regardless of CLI order, and every override is built via `dataclasses.replace` on a fresh `Settings()`, never mutating the frozen base. (spec 0016)
- The OQ2 recommender refuses to calibrate over a gate it has measured to be broken: `EvalConfig.gate_false_escalation_ceiling` (eval-only, `Settings`-disjoint, provisional `0.20`) is the gate-confound bar, and `recommend_oq2(points, measured_false_escalation, eval_config)` is a dispatcher wrapping the unchanged `rank_sweep` â€” a measured instruct-judge false-escalation **strictly `> ceiling`** returns the `gate-confounded` typed null (`OUTCOME_GATE_CONFOUNDED`, carrying the measured rate) rather than tuning `verify_threshold` over a judge that rejects correct citations; `== ceiling` is not confounded (strict `>`) and an unmeasured `None` defers. `Recommendation.outcome`/`.gate_false_escalation_measured` are additive (appended last-with-defaults). (spec 0019)
- `preflight_models_present(settings, tags_payload, *, resolver=None)` is a setup-time doctor (`PreflightError` + `cmd_preflight` + a `preflight` CLI subparser): it asserts `gateway.assert_local` on the loopback endpoint **first** (no second outbound path â€” the `/api/tags` read is the same loopback-gated egress class it preflights), then checks a **deduped required-tag set** (`_required_model_tags`: `scout_model` âˆª `lm_model`, since deep-model and the instruct judge both resolve to `lm_model`) is served, raising a typed error **naming** the first absent tag. It claims only that models are **pulled**, NOT co-resident-loadable â€” OOM under `mode=auto` is named as a residual mid-run risk the cheap G1 smoke catches (no-false-capability). (spec 0019)
- The eval report schema is `0014/1` (additive last-with-defaults over `0013/1`): run-metadata `gate_false_escalation_ceiling`; aggregate `gate_confounded`/`gate_confounded_measured_rate` + instruct/scout A/B false-escalation twins (`gate_false_escalation_{instruct,scout}` + `_count`/`_total`, null-with-zero-count), hoisted to one `_GATE_CONFOUND_AGG_FIELDS` anti-drift source with a drift-guard test; legacy `0013/1` blocks still validate. `run_swebench_sweep` wires `recommend_oq2` in (best-achievable instruct false-escalation = min over measured grid points, only when the base `verify_method == "instruct_model"`) and stamps the ceiling into metadata + `outcome`/`gate_false_escalation_measured` into the recommendation block â€” the gate-confound outcome is reached by the real driver, not left wired-but-dormant. (spec 0019)
- The OQ2 G3 outcome taxonomy is a **projection layer above the byte-frozen `recommend_oq2`**: `classify_g3_outcome(recommendation, aggregate, eval_config)` (in `harpyja/eval/oq2_classify.py`) maps to exactly one of {`RECOMMENDATION`, `GATE_CONFOUNDED`, `DEGRADED_DOMINATED`, `NOT_SEPARABLE`} by the total order `DEGRADED_DOMINATED > GATE_CONFOUNDED > NOT_SEPARABLE > RECOMMENDATION`, recording ALL true blocking conditions (not just the winner). The no-survivor signal `S = (incumbent_validated is False AND advantage_exceeds_variance is False)` is read from the frozen `Recommendation` and computed only when the outcome is not `gate-confounded` (else n/a â€” no phantom `NOT_SEPARABLE`); `indicative_only` is a `RECOMMENDATION`-only sub-flag (effective_N < `n_floor`). The dispatcher/`rank_sweep` stay byte-frozen, pinned by a field-reachability lock and a behavior snapshot. (spec 0020)
- The OQ2 operator run is a sequential G0â†’G1â†’G2â†’G3 stop-and-report protocol (`harpyja/eval/oq2_protocol.py`, driven live by `oq2_live.py` + the `oq2` CLI) that records each gate verdict before the next runs and emits a NEW pinned **gate-ledger** (`LEDGER_SCHEMA_VERSION = "0020/1"`, distinct from report `0014/1`, loudly validated, written via the shared `atomic_write_json`). Close-vs-hold is drawn **by cause**: G0 preflight fail, fixtures absent, or a G1 sub-check-(a) non-completion for an environment reason (OOM/resource) â†’ `BLOCKED` **hold**; a run that completed then degrade-dominated or gate-false-rejected â†’ `STOP:SMOKE` **close**; a G3 label closes. A measurement spec closes only on a recorded SUT-observing typed outcome â€” skip-not-fail because the environment is absent is **never** a valid close. (spec 0020)
