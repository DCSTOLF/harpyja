# Spec 0042 — adoption

Symbols-tool adoption — advertise, sell, reposition, fix the plumbing.

## Why

0/28 `symbols` invocations across all three models (0040 pilot) is OVERDETERMINED, not model disinterest — the audit found four stacked defects:

1. **Stale prompt advertisement.** `build_initial_prompt` (`context_map.py`, written in 0027 when the suite was 4 tools) enumerates only "ls, glob, and grep" and was NEVER updated when `symbols` landed in 0030 — nor does it name `read_span`. For an instruction-following 4–14B, that list IS the menu.
2. **Description says what, not when.** The tool description says what `symbols` returns but not WHEN to use it, nor that results carry exact start/end line spans per symbol — i.e. citation-shaped output, the fastest path from candidate-file to the `file:line` that `submit_citations` wants.
3. **Positioning gap.** `path` is REQUIRED, so `symbols` is only reachable AFTER a candidate file is found — by which point grep already produced line numbers, so the model perceives no marginal value.
4. **Result-shape penalty.** `symbols` alone returns a nested dict `{"symbols": [...], "degraded": bool}` while every other nav tool returns `list[CodeSpan]` — and `_spans_of()` (`explorer_loop.py`) unwraps only lists and `{"path": ...}` mappings, so a `symbols` result registers ZERO spans in seen-span/loop-detection bookkeeping: a repeated `symbols` call reads as an unproductive repeat and its locations never enter new-span accounting. The tool is structurally PENALIZED for being called.

CONSEQUENCE: 0030's "lift unproven" and every subsequent "symbols not invoked" (0031/0034/0035/0040) measured an UNUSABLE tool, not an unhelpful one. This spec gives it a fair test. It targets 14b's dominant near-miss (right-file-wrong-span) — the exact failure `symbols` exists to fix.

Ref: 0030 (symbols tool + refuted lift), 0027 (the stale prompt), 0035 (marker convention), 0040 (0/28 adoption, RFWS-dominant pilot), 0041 (exclusive-endpoint gate + gated driver — the clean-measurement prerequisite).

**INVARIANT (fix all four, or the test is unfair again):** prompt-advertisement, when-to-use description, result-shape parity, and the positioning gap are all fixed. Fixing three and re-measuring would produce another uninterpretable null. **Accepted attribution confound (recorded so no follow-up re-litigates it):** because all four fixes land at once, a positive adoption result cannot attribute WHICH fix drove adoption — that is a deliberate trade: fairness of the test over attributability of the fix.

**INVARIANT (prompt↔tool-suite must be structurally bound):** the stale prompt is the root defect and it drifted silently for 5 specs because nothing asserted it. The prompt's tool enumeration is DERIVED from (or asserted equal to) the registered tool list — the exact-tool-count convention (3→4→5 hard-count tests) extended to the PROMPT, which the count tests never covered. A new tool must be un-shippable without appearing in the prompt.

**INVARIANT (measure the fix, don't assume it):** the deliverable is a re-measured adoption rate + RFWS→exact conversion on the pilot cases, run through the 0041-gated driver with committed artifacts. Honest null is a valid outcome: if adoption rises but buckets don't improve at sufficient N, the tool's value is genuinely refuted — THIS time by a fair test.

**INVARIANT (no capability claim beyond the measurement):** pilot N is small; report adoption + conversion, do NOT declare the tool "proven." The RFWS→exact denominator at pilot N is a handful of cases, so a no-conversion outcome below the pre-registered power floor is typed as under-powered, NOT as refutation. Model spread the measurement may generate is a bake-off input, not a ranking.

**INVARIANT (frozen predicate before live spend — the 0023/0026/0039/0040 discipline):** every typed-outcome boundary in AC7 is a total pure function over a pre-registered, frozen, hashed config committed BEFORE any live call, computed from retained per-case paired buckets against the committed 0040 pilot ledger — never marginal counts.

## What

1. **PROMPT:** name `symbols` AND `read_span` in `build_initial_prompt` (both are absent today); state when to use `symbols`. Add the structural assertion binding the prompt's enumeration to the registered tool list (drift-proof). The asserted set is DERIVED from the same single source the existing `test_tool_schemas_match_the_built_tool_surface_single_source` test uses — no second competing definition of "the registered tool list." The assertion covers the FULL registered surface: the navigation tools `{grep, glob, read_span, ls, symbols}` and the terminal `submit_citations` (documented in the prompt as terminal, distinct from navigation).
2. **DESCRIPTION:** state the when-to-use and that results carry exact start/end spans per symbol — the citation-shaped-output pitch, explicitly aimed at candidate-file → precise-span (the RFWS failure).
3. **RESULT SHAPE:** `symbols` returns a bare span list like every other nav tool — `list[CodeSpan]` in the clean case. The `degraded: bool` dict field is REPLACED by a stable degraded-marker string PREPENDED as a list element (`list[str | CodeSpan]`, marker first): `_spans_of()` unwraps the list and counts ONLY the `CodeSpan` entries (spans enter seen-span/loop-detection accounting; the marker never counts as a span), while result stringification keeps the marker MODEL-VISIBLE — preserving 0030's never-a-silent-downgrade contract. This deliberately EXTENDS the 0035 marker convention from "unsearchable-scope replacement" to a second, distinct case — "successful-but-degraded annotation" — and the 0035 convention text is amended in lockstep to record both cases.
4. **POSITIONING (the adoption driver):** make repo-wide symbol lookup reachable — `path` optional, or a `find_symbol(name)` affordance backed by Tier-0's existing pre-extracted symbol records. This is where the tool earns adoption AND is a partial answer to lexical-unreachability (astropy: "separability matrix" can't be grepped to `separable.py`, but a repo-wide symbol lookup for `separability_matrix` finds it BY NAME). Read-only, repo-confined, output-clamped like every tool. When Tier-0 symbol records are absent or the index is degraded, the repo-wide lookup returns a TYPED marker in the 0035 REPLACEMENT shape (a bare marker string replacing the result — no spans exist to annotate, unlike What #3's file-local degraded case, which is an ANNOTATION on real spans) — never a silent `[]` indistinguishable from "no such symbol."
5. **PRE-REGISTERED MEASUREMENT CONFIG:** a frozen, hashed `PREREGISTERED_ADOPTION_CONFIG_0042` committed before any live call, pinning: the adoption boundary (STILL_NOT_ADOPTED iff clean-cell `symbols` invocations = 0 — the baseline is exactly 0/28, so any nonzero clean-cell use disproves "structurally unreachable"; adoption RATE is reported alongside, never re-thresholded post-hoc); the RFWS denominator (the committed 0040 clean cells bucketed RFWS, per model); the conversion predicate (per-case paired buckets — 0040-ledger bucket vs re-run bucket per case — computed from retained pairs, never marginals, with BOTH flip directions retained and reported: RFWS→exact conversions AND exact→RFWS regressions, the net surfaced alongside the conversion count so a single noise flip over a net-negative re-run cannot type CONVERTS unqualified); the power floor `MIN_RFWS_DENOMINATOR` below which a no-conversion result is typed under-powered rather than refuting; the model coverage the closure run must include (pinned pre-run, so "wall-clock allows" cannot quietly shrink the measurement); and the adoption denominator under partial model coverage (per-model boundaries — a 14b-only run is typed against 14b's own clean-cell universe, never against the pooled 0/28 baseline it never had).
6. **GATED RE-MEASUREMENT + CLOSURE ARTIFACTS:** the re-measure runs through `run_gated_pool_pilot` (or an equivalently 0041-gated driver) with the `0041/pilot/2` exclusivity proof in the ledger, on the committed 0040 pinned pilot case set (consumed, not re-selected), via a STOP-AND-WARN operator driver. Durable per-case trajectory-verified artifacts (model identity, tools invoked — including per-case `symbols` invocation count — terminal bucket, exclusivity proof), or a versioned aggregate derived from them, are committed before the spec closes.

## Acceptance criteria

## Open questions

1. Repo-wide lookup shape: `path` optional on the existing `symbols` tool, vs a distinct `find_symbol(name)` tool (which changes the tool count 5→6 and the exact-count convention). Lean optional-path (no new tool, less menu bloat for a small model) — but a distinct name may advertise the capability better to a 4B. Interaction with AC1: if optional-path wins, the name-presence drift guard alone cannot verify the repo-wide capability is ADVERTISED — add a when-to-use presence check for the repo-wide affordance specifically. Decide before plan.
2. Output clamp for repo-wide lookup: a common name could match hundreds of symbols across a big repo. Clamp hard and rank with the ranking PINNED (exact-name > prefix > substring) in frozen config or a tool-contract test, so it's not arbitrary truncation. Consider a distinct clamp knob for repo-wide lookup vs the file-local `scout_symbols_max_entries=400` — the blast radius of a common name across a repo differs from one file.
3. Does the when-to-use text belong in the tool description (schema) or the initial prompt (or both)? Small models weight the enumerated prompt list heavily; the schema description may be under-read. Lean both, with the prompt carrying the when.
