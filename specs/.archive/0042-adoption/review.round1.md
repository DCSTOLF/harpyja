## Convergent findings

**1. Degraded-marker vs. bare-`list[CodeSpan]` shape contradiction (What #3 / AC2).**
Both agents independently flagged this as self-contradictory as written. Codex: "the result-shape requirement is internally ambiguous: `symbols` must return bare `list[CodeSpan]`, but `degraded` must also become a marker. A marker string would break the bare-list claim; a marker embedded in `CodeSpan` needs to be specified" — and filed no separate convention violation here but treated it as a blocking concern. Claude-p went further and filed it as a **convention violation**: the 0035 marker convention is specifically the *unsearchable-scope* branch of a three-way split (a bare string *replacing* the result); a degraded symbols result is different in kind — a successful result (ripgrep def-scan spans) that must *also* carry a visible degraded flag, and 0030 explicitly required the fallback stay visible, never a silent downgrade. Both agents converge on needing a concrete decision (marker embedded in an existing per-span field vs. a separate model-visible note vs. a deliberately widened return contract with all consumers updated) and, if 0035's marker semantics are being extended to a new "successful-but-degraded" state, amending the 0035 convention text itself.

**2. AC5's measurement closure is not strong enough to close a measurement spec.**
Codex: "AC5 makes the live re-measurement an integration skip-not-fail path, but the spec's deliverable is a recorded adoption/conversion outcome. A skipped live test cannot close a measurement spec" — filed as a **convention violation** ("a measurement spec closes on a recorded, SUT-observing typed outcome; skip-not-fail is never a close"), alongside a second **convention violation** that the live measurement doesn't explicitly require the trajectory-verified artifact convention (model identity, tools invoked, terminal bucket, exclusivity proof). Claude-p converges from a different angle: AC6's typed outcome has no pre-registered decision predicate — under the repo's frozen-config discipline (0023/0026/0039/0040) "the verdict must be a total pure function over a committed config," and the ADOPTED vs. ADOPTED_AND_CONVERTS/ADOPTED_NO_CONVERSION boundary must be pinned, hashed, and committed *before* the run, computed from per-case paired buckets against the committed 0040 pilot ledger, never marginal counts — filed as a **convention violation** of the same frozen-config discipline. Claude-p additionally suggests AC5 explicitly bind to `run_gated_pool_pilot` (or an equivalently gated driver) with the `0041/pilot/2` exclusivity proof in the ledger, rather than "a 0041-gated exclusive endpoint" in prose. Both agents, from different angles (closure mechanics vs. predicate freezing), agree AC5/AC6 as written cannot mechanically and honestly close the spec's core deliverable.

**3. The prompt-bound tool set in AC1 is underspecified.**
Codex: "the prompt/tool-suite binding is underspecified around which registered tools are in scope: navigation tools only, or also the terminal `submit_citations` action. That ambiguity can produce a false drift guard" — and suggests naming the exact set explicitly, e.g. `{grep, glob, read_span, ls, symbols}` for navigation, documenting `submit_citations` separately as terminal. Claude-p converges independently: derive AC1's assertion from the same single source the existing `test_tool_schemas_match_the_built_tool_surface_single_source` test uses, so there aren't two competing definitions of "the registered tool list"; decide whether `submit_citations` and `read_span` are in the asserted set (note `read_span` is also absent from the current prompt enumeration and should be promoted from parenthetical to explicit scope in What #1).

## Divergent findings

- **AC3's undefined degrade path when Tier-0 symbol records are absent** (claude-p only). `wiring.py:50` already tolerates `load_symbols_or_none(...)` or `[]`; a repo-wide `find_symbol` over an empty record set needs a defined honest outcome (typed marker or clamped ripgrep def-scan fallback), or the graceful-degradation guardrail is met by accident rather than design. Codex did not raise this. Judgment: valid and concrete gap directly tied to the spec's own "read-only, repo-confined, output-clamped" contract for AC3 — should be treated as a required revision, not merely advisory.

- **AC5/AC6 statistical power at pilot N** (claude-p only). The RFWS→exact conversion denominator on 14b's dominant-RFWS subset is a handful of cases (claude-p estimates ~3-5), so ADOPTED_NO_CONVERSION's parenthetical ("value refuted by a fair test") can overclaim refutation from what may be 0-of-a-few conversions. This is distinct from codex's closure-mechanics concern (item 2 above) — it's about the honesty of the outcome label itself, not whether the run happens. Codex did not raise this. Judgment: valid, consistent with the spec's own "no-capability-claim beyond the measurement" invariant, which only partially covers it — should be a required wording fix to AC6, not merely advisory.

- **Attribution confound from fixing all four defects simultaneously** (claude-p only). Deliberate per the fairness invariant, but the spec should say explicitly that a positive result cannot attribute which fix drove adoption, so a follow-up doesn't re-litigate it. Codex did not raise this. Judgment: low-risk, documentation-only — advisory.

- **OQ1 interaction with AC1's drift-guard** (claude-p only). If optional-`path` wins for repo-wide lookup, the name-presence drift guard cannot verify the repo-wide capability is *advertised* — which is the whole positioning point of What #4. Codex did not raise this. Judgment: valid, but the spec already correctly defers OQ1 to a pre-plan decision; this is guidance for that decision, not a new gap — advisory.

- **OQ2 ranking/clamp guidance** (claude-p only): pin the ranking (exact-name > prefix > substring) in frozen config or a tool-contract test, and consider a distinct clamp knob for repo-wide lookup vs. the existing file-local `scout_symbols_max_entries=400`. Codex did not raise this. Judgment: reasonable refinement of an already-open question — advisory.

- **AC6 baseline framing** (claude-p only): record that the 0/28 baseline was measured under the old stale-prompt/wrong-shape condition, so the reported delta is fix-vs-defect, not tool-vs-no-tool. Codex did not raise this. Judgment: good documentation hygiene, low risk — advisory.

No item was raised by codex without a claude-p counterpart or explicit acknowledgment above; all codex concerns, suggestions, and convention violations are captured in the Convergent section and the Required revisions below.

## Required pre-plan revisions

1. **Resolve the degraded-marker shape contradiction in What #3 / AC2.** Decide and specify the exact representation (e.g., a stable marker/sentinel embedded in an existing per-`CodeSpan` field, a separate model-visible note alongside the list, or a deliberately widened return contract with every consumer — including `_spans_of()` — updated in lockstep). If 0035's marker semantics are being extended from "unsearchable-scope replacement" to "successful-but-degraded," amend the 0035 convention text itself to reflect the new case.

2. **Freeze the AC5/AC6 decision predicate before the live run.** Pin, as a committed/hashed config per the 0023/0026/0039/0040 frozen-config discipline: what adoption rate/count counts as ADOPTED; the per-case paired-bucket definition (against the committed 0040 pilot ledger, never marginal counts) that separates ADOPTED_AND_CONVERTS from ADOPTED_NO_CONVERSION; and which pilot cases constitute the RFWS denominator.

3. **Strengthen AC5's closure path.** Replace the bare "skip-not-fail integration test" framing with an explicit closure mechanism: bind the re-measurement to `run_gated_pool_pilot` (or an equivalently gated driver) with the `0041/pilot/2` exclusivity proof in the ledger; add a spec-local STOP-AND-WARN operator-driver requirement; and require durable, trajectory-verified per-case artifacts (model identity, model invoked, tools invoked, terminal bucket, exclusivity proof) — or a versioned aggregate artifact derived from them — committed before the spec can close.

4. **Name the exact prompt-bound tool set in AC1.** State explicitly whether the drift-guard covers navigation tools only or also the terminal `submit_citations` action; derive the assertion from the same single source as the existing `test_tool_schemas_match_the_built_tool_surface_single_source` test; and promote `read_span` from a What #1 parenthetical to explicit, asserted scope.

5. **Define AC3's honest outcome when Tier-0 symbol records are absent or degraded.** Specify a typed marker or a clamped ripgrep def-scan fallback for the empty-index case, rather than allowing a silent `[]` that reads indistinguishably from "no such symbol."

6. **Add an under-powered qualifier / explicit N floor to AC6's ADOPTED_NO_CONVERSION label.** The RFWS→exact conversion denominator at pilot N is small; the outcome text should not read as an unqualified refutation of the tool's value when the underlying count may be 0-of-a-few.

## Advisory (may resolve at plan time)

- State explicitly in the spec that fixing all four adoption-blocking defects simultaneously (deliberate, per the fairness invariant) means a positive result cannot attribute which individual fix drove adoption, so a future spec doesn't need to re-litigate this confound.
- For OQ1: note the interaction with AC1 — if optional-`path` wins for repo-wide symbol lookup, the name-presence drift guard alone cannot verify that the repo-wide capability is *advertised*; add a when-to-use presence check for that affordance specifically.
- For OQ2: pin the ranking (exact-name > prefix > substring) in frozen config or a tool-contract test, and consider a distinct output-clamp knob for repo-wide lookup versus the existing file-local `scout_symbols_max_entries=400`, since the blast radius of a common name across a whole repo differs from within one file.
- In AC6, record that the 0/28 baseline was measured under the old stale-prompt/wrong-result-shape condition, so the reported adoption delta is properly framed as fix-vs-defect rather than tool-vs-no-tool.

## Verdict summary

Both reviewers independently verified the spec's diagnosis against source and found the four-stacked-defect argument, the fairness invariant, and the out-of-scope fence sound — this is the strongest part of the spec and needs no rework. Quorum is met via claude-p's approve-with-comments, so the spec proceeds to `reviewed`, but codex's changes-requested concerns are not superseded by that quorum: both agents converge, from different angles, on the same two blocking gaps — the degraded-marker/bare-list shape contradiction in What #3/AC2, and a measurement-closure weakness in AC5/AC6 (codex: skip-not-fail cannot close a measurement spec and lacks a trajectory-verified-artifact requirement; claude-p: the typed outcome lacks a frozen, pre-registered decision predicate and an explicit gated-driver binding) — plus a shared concern that AC1's prompt-bound tool set is ambiguously scoped. These three convergent items, together with claude-p's additional required fixes (AC3's undefined empty-index degrade path and AC6's under-powered-outcome qualifier), must be resolved in the spec text before `/speccraft:spec:plan` is run, even though the spec formally advances to `reviewed` now.
